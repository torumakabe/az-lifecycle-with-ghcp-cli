# ラボ1: 調査と設計判断、記録
**テーマ**: 調査から設計判断の記録まで

## 前提条件

- [ラボ環境セットアップガイド](./setup/README.md)を完了していること
- `docs/adr/` に既存 ADR がない状態（クリーンスタート）

## シナリオ

新しい Python Web アプリを設計する。
まず Deep Research でサービス選定の調査を行い、要件に合うAzureのコンピューティングとデータベースサービスを決定する。
次に、選定したサービスの推奨構成を調査し、調査レポートとして保存する。
最後に、調査から得られた設計判断を ADR として記録する。

「サービス選定 → 構成調査 → 判断記録」までをターミナルで完結させるワークフローを体験する。

---

## Part 1: Deep Research — 技術調査

### ステップ1: サービス選定の調査

要件に合う Azure サービスを選定するための調査を行う。`/research` で要件を伝え、候補の比較と推奨を依頼する。

> [!TIP]
> **MCP Server の活用**
>
> `/research` による調査では、Azure MCP Server や MS Learn MCP Server が公式ドキュメントの参照に活用される。これらの MCP Server は前提条件のセットアップで設定済みであり、エージェントがサービス仕様・価格・制約などの最新情報を正確に取得するための基盤となる。

**Copilot CLIに入力:**

```
/research 次の要望を満たす Azure サービスの組み合わせを調査し、推奨を提示して
- Python の Web API アプリケーション
- Python のバージョンは 3.13
- フレームワークは FastAPI
- Python のパッケージ管理は uv
- ピーク時のリクエスト数は100/秒程度
- RDBMSを使う（トランザクションあり）
- コンテナ技術（Docker等）の導入予定はなく、コードを直接デプロイしたい
- チームに PostgreSQL の経験があり、既存の運用知識を活かしたい
- パスワードレス認証（Entra ID）でデータベースに接続したい
- リージョンは東日本
- コンピューティングサービスとデータベースサービスそれぞれについて、候補を比較したい
- SKU やプランは未決定でよい
結果は docs/research/azure-service-selection.md に保存して
```

**期待する結果:**

- Azure MCP Server / MS Learn MCP Server 等を活用した Deep Research が実行される
- コンピューティングサービスとして App Service / Container Apps / AKS 等が比較される
- データベースとして PostgreSQL Flexible Server / MySQL Flexible Server / Azure SQL Database 等が比較される
- 要件との適合度に基づき、**App Service** と **Azure Database for PostgreSQL Flexible Server** が推奨される（2026/4時点ではそうなるはず）
- 調査レポートが `docs/research/azure-service-selection.md` に保存される

---

### ステップ2: 選定結果の確認と ADR 作成

調査レポートを確認し、推奨されたサービスの選定理由を検証する。納得できれば、サービス選定を ADR として記録する。

> [!TIP]
> `/research` は Web 検索を含む広範な情報収集を行うため、最新のサービス仕様や比較情報を踏まえた推奨が得られる。ただし、最終的な選定判断は人間が行う。調査レポートの比較表や理由を確認し、要望に照らして納得できるか検証すること。

**期待する結果:**

- 推奨サービスとその理由が明確に記載されている

推奨に合意できたら、サービス選定の ADR 作成を指示する。

**Copilot CLIに入力:**

```
App Service と PostgreSQL Flexible Server を採用する。サービス選定の判断を ADR として記録して。コンテキストは @azure-service-selection.md を参考にして
```

> [!TIP]
> **判断したら記録**
>
> サービス選定は典型的な ADR 候補。保存しないと選定の文脈（なぜ Container Apps でなく App Service か等）が失われてしまう。

> [!TIP]
> **`@` によるファイル参照**
>
> プロンプト中で `@` に続けてファイル名を入力すると、ワークスペース内のファイルをインクリメンタル検索、パス補完できる。選択したファイルはコンテキストとして添付される。

ADRは以降の開発、運用で重要な情報源になる。吟味し、必要に応じて編集する。

**期待する結果:**

- `manage-adr` エージェントが起動し、サービス選定の ADR が作成される
- コンピューティングサービスとデータベースサービスそれぞれの ADR が `docs/adr/` に作成される
- 却下した代替案（Container Apps、Azure SQL 等）とその理由が明記されている
- `docs/adr/INDEX.md` の一覧テーブルも作成される

---

### ステップ3: 推奨構成の調査

サービスが決まったら、具体的な構成を調査する。サービス選定とADR作成を通じて生じた要望があれば添える。

> [!TIP]
> **Skill・MCP Server・フックの整備**
>
> 選定した技術に関する一般知識の参照先（MCP Server 経由の公式ドキュメント等）や、プロジェクト固有のルール（Skill、`copilot-instructions.md`）は、この調査段階で整備しておくと、以降のコード生成やレビューが期待に沿うものになりやすい。また、postToolUse フックを設定しておくことで、生成コードの品質を機械的に維持できる。このラボでは Azure MCP Server・MS Learn MCP Server（ステップ1で紹介済み）、`azd-infra-rules` Skill（azd 前提 IaC ルール）、postToolUse フック（ruff・az bicep build の自動実行）を事前に整備済みである。

**Copilot CLIに入力:**

```
/research @docs/adr/ と次の要望を満たす推奨構成を調査して
- App Service と PostgreSQL の間は常に VNet + Private Endpoint 経由のプライベート接続とする（PoC・本番共通）
- データベースは本番環境ではインターネットに公開しない
- ただし PoC 段階では開発者が手元のツール（psql 等）から直接接続できるよう、DB パブリックアクセスも有効にする
- PoC と本番の構成が大きく変わらないようにしたい（もちろんサイズや SKU は違ってもよい）
- PoC で性能の基礎値を測りたいため、PoC と本番は同じプランレベルとする。PoC のサイズは最小とする
- Azureサービスのログとメトリクス、アプリケーションのトレースを収集
- アプリケーションのトレースは自動計装
- ログとメトリクス、トレースは取り込みコストが低くなるように。保存期間は既定
- Azure Developer CLI（azd）でデプロイする前提
結果は docs/research/recommended-architecture.md に保存して
```

**期待する結果:**

- `docs/research/recommended-architecture.md` に調査レポートが作成される
- Azure MCP Server / MS Learn MCP Server 等を活用した Deep Research が実行される
- 選定したサービスの、より踏み込んだ推奨構成レポートが得られる

> [!IMPORTANT]
> - 構造化された詳細なレポートが保存されているか確認する（内容はモデルや参照ソースにより異なる）
> - 複数ソースからの情報が統合されているか確認する

---

## Part 2: 設計判断と ADR — 判断の記録

### ステップ4: 構成に関する設計判断の抽出と ADR 作成

構成調査の結果から、記録すべき設計判断を抽出する。サービス選定の ADR はステップ2で作成済みなので、ここでは構成レベルの判断（認証方式、ネットワーク構成等）に集中する。抽出の基準は「6ヶ月テスト」— 6ヶ月後に誰かが「なぜこうした？変えていい？」と聞くかどうか。

**Copilot CLIに入力:**

```
@recommended-architecture.md の調査結果から、ADR にすべき設計判断を洗い出して。判断基準は「6ヶ月後に別の人が『なぜ？変えていい？』と聞くか」で。サービス選定の ADR は作成済みなので、構成レベルの判断に絞ること
```

提案されたADRに合意できれば、作成を指示する。 `docs/adr/` に作成される。

**期待する結果:**

- 調査レポートが読み込まれ、構成レベルの設計判断が抽出される
- 各判断に対して「なぜ ADR にすべきか」の理由が示される
- SKU 選定など一時的な判断は「ADR 不要」としてスキップされる
- ADR 候補の確認後「はい」と返答すると、`manage-adr` エージェントが起動し ADR が一括作成される
- 各 ADR が30行以内に収まっている
- 却下した代替案とその理由が明記されている
- `docs/adr/INDEX.md` の一覧テーブルが更新される

> [!TIP]
> 洗い出しと作成は1つのプロンプトから連続して行われる。`manage-adr` エージェントが判断抽出から ADR 作成・INDEX 更新までを一貫して処理するため、ステップを分ける必要がない。ADR は30行以内でスリム — コーディングエージェントが `docs/adr/` を読み込む際の負担を最小化する設計。

---

## まとめ

- **`/research` でサービス選定** → 要件に基づくサービス比較と推奨を得る
- **判断したら即 ADR** → サービス選定を `docs/adr/` に記録
- **`/research` で構成調査** → 選定したサービスの推奨構成を調査し `docs/research/<topic>.md` に保存
- **構成の設計判断も ADR として記録** → `docs/adr/` に追加
- **ADR は30行以内** — フォーマットは `docs/adr/README.md` に定義
- **調査 → 判断 → 記録まで、ターミナルで完結**
