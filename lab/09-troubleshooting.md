# ラボ9: トラブルシューティング
**テーマ**: 障害対応の初動を格段に速くする

## 前提条件

- [ラボ環境セットアップガイド](./setup/README.md)を完了していること
- ラボ7（デプロイ）が完了し、Bicep で定義したリソースがデプロイ済みであること
- ラボ6（検証用アプリ）の検証アプリがデプロイ済みであること
- App Serviceに意図的な設定ミスを仕込み済み（下記「事前準備」参照）

## 事前準備: 意図的なエラーの導入

> [!NOTE]
> ラボ開始前に以下を実行し、検証用アプリが500エラーを返す状態にしておく。

ラボ7でデプロイした検証用アプリは、Entra ID パスワードレス認証で PostgreSQL Flexible Server に接続する。ここで意図的にアプリ設定を変更し、接続先ホスト名を間違った値に書き換えることで500エラーを発生させる。

まず、App Service に設定されているアプリケーション設定の一覧を確認し、PostgreSQL 接続先ホスト名に使われている環境変数名を特定する。

**Copilot CLIに入力（環境変数名の確認）:**
```
デプロイ済みのApp Serviceのアプリケーション設定一覧を表示して。PostgreSQLの接続先ホスト名に使われている環境変数名を教えて
```

**期待する結果:**
- Azure MCP Server の `appservice` ツールでアプリケーション設定が取得される
- PostgreSQL ホスト名の環境変数名（例: `POSTGRES_HOST`）と現在の値が表示される

確認した環境変数名を使い、接続先を間違った値に書き換える。

**Copilot CLIに入力（設定変更）:**
```
デプロイ済みのApp Serviceのアプリケーション設定 <確認した環境変数名> を wrong-server.postgres.database.azure.com に変更して
```

> [!TIP]
> `<確認した環境変数名>` は前のステップで特定した実際の環境変数名に置き換えること。環境変数名を間違えると、新しい設定が追加されるだけでアプリの動作に影響しない。

## シナリオ

ラボ7でデプロイした検証用アプリ（ラボ6）で500エラーが発生している。Copilot CLI を使って原因を調査し、修正する。

---

## ラボ手順

### セッションのクリア

事前準備で Copilot CLI が設定変更を行ったため、セッションに原因のコンテキストが残っている。トラブルシューティングの実力を試すために、セッションをクリアする。

**Copilot CLIに入力:**
```
/clear
```

> [!TIP]
> `/clear` でセッション履歴がリセットされる。Copilot CLI は事前準備の内容を知らない状態から調査を始めることになる。

---

### ステップ1: 状況の確認

**Copilot CLIに入力:**
```
デプロイした App Service の /check-db エンドポイントへのアクセスでエラーが出ているらしい。状況を確認して
```

**期待する結果:**
- Azure MCP Server の `appservice` ツールなどで App Service の情報が取得される
- アプリのエンドポイント（`/health`、`/check-db` など）への疎通確認が行われる
- エラーの原因（例: PostgreSQL の接続先ホスト名が `wrong-server.postgres.database.azure.com` になっており DNS 解決に失敗）が特定される

> [!TIP]
> Copilot CLI は App Service の設定取得、エンドポイントへの HTTP リクエスト、エラーメッセージの分析などを組み合わせて、原因特定を支援してくれる。

---

### ステップ2: 修正と確認

原因がわかったら、修正と動作確認を依頼する。

> [!NOTE]
> 前のステップで、先回りして修正まで提案されることがある。そのタイミングで受け入れてもよい。

**Copilot CLIに入力:**
```
原因が分かれば修正して、動作確認もして
```

**期待する結果:**
- `POSTGRES_HOST` が正しい値に修正される（`az webapp config appsettings set` の実行、または Azure MCP Server 経由）
- 修正後、`/health` や `/check-db` エンドポイントへの疎通確認が行われる
- PostgreSQL への接続が成功し、エラーが解消されたことが確認される

---

## まとめ

- Copilot CLI は、調査 → 原因特定 → 修正 → 確認を支援してくれる
- Azure MCP Server による設定取得、HTTP リクエストによるエラー再現、原因分析を Copilot CLI が自律的に組み合わせる
- 従来のように Azure Portal でログを探し、ドキュメントを調べ、CLI コマンドを手打ちする手順と比較してみよう
