# ラボ6: 検証用アプリ生成
**テーマ**: IaCの設計に対応した検証用アプリを自動生成し、ADRに記録された設計判断がコードに自然と反映されることを確認する

## 前提条件

- [ラボ環境セットアップガイド](./setup/README.md)を完了していること
- ラボ2（IaC生成）が完了していること

## シナリオ

ラボ2で作成した Bicep は「App Service + PostgreSQL Flexible Server（Entra ID パスワードレス認証）」の構成を定義している。デプロイ前に「本当に Entra ID 認証で接続できるのか？」「レイテンシは許容範囲か？」を確認したい。ここでは Copilot CLI を使って検証用アプリを生成し、設計の妥当性を事前に検証できるようにする。

ポイントは、`copilot-instructions.md` が `docs/adr/` の設計判断を参照していること。これにより、プロンプトで個別のライブラリや設定を指示しなくても、ADR の判断がコード生成に反映される。

このラボではコード生成とレビューまでを行い、実際のデプロイと動作確認は[ラボ7](./07-deploy.md)で `azd up` と合わせて実施する。

---

## ステップ1: 検証用アプリの生成を依頼

**Copilot CLIに入力:**
```
@infra/ の構成に対して、以下を検証するシンプルなWebアプリをPython 3.13 + FastAPIで作って:

1. ヘルスチェックエンドポイント（/health）
2. DB接続確認エンドポイント（/check-db） — 認証方式は infra/ の定義に従う
3. レスポンスタイム計測（各エンドポイントで処理時間をms単位で返す）

src/verification-app/ に配置して
```

**期待する結果:**
- `infra/` 配下の Bicep ファイルが読み込まれ、構成が解析される
- 検証用アプリが `src/verification-app/` に生成される

---

## ステップ2: 生成されたコードの確認（ADR準拠チェック）

**Copilot CLIに入力:**
```
/review @src/verification-app/ をレビューして。特に @docs/adr/ の設計判断が正しく反映されているかと、 @infra/ @azure.yaml @hooks/ との統合ができているか。2つのサブエージェント（Opus 4.6 と GPT 5.4）で実行して
```

レビューで提案された改善点が妥当であれば、受け入れる。

**期待する結果:**
- コードに問題がないか客観的にレビューされる
- ADR の設計判断がコードに正しく反映されているか、IaC との統合ができているかがレビューされる

---

## まとめ

- **ADR の設計判断とIaCがコード生成に反映される** — プロンプトに個別指示しなくても ADR と IaC に従ったコードが生成される
- **実際のデプロイと動作確認は[ラボ7](./07-deploy.md)で実施**
