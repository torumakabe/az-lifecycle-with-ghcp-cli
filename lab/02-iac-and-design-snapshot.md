# ラボ2: IaCコード生成とDesign Snapshot — ADRからBicepへ
**テーマ**: ADRに基づくIaC生成と、アーキテクチャ概要のスナップショット出力

## 前提条件

- [ラボ環境セットアップガイド](./setup/README.md)を完了していること
- ラボ1が完了し、以下が作成済み:
  - `docs/adr/` に ADR（ラボ1で作成されたもの）

## シナリオ

ラボ1で作成した ADR を入力として、Bicep テンプレートを生成する。
調査レポートと ADR の設計判断が IaC コードに反映されることを確認する。
最後に、IaC と ADR からアーキテクチャ概要説明書をスナップショットとして自動生成する。

---

## Part 1: IaC 生成 — ADR から Bicep へ

### ステップ1: 実装プランの作成

Plan モード（`Shift+Tab` → Plan を選択）に切り替えてから入力する。

**Copilot CLIに入力:**

```
@docs/research/recommended-architecture.md と @docs/adr/ をもとに bicepコードを生成して。azd 構成（azure.yaml、hooks）も含めて
```

**期待する結果:**

- `docs/research/recommended-architecture.md` と `docs/adr/` が読み込まれ、推奨構成と設計判断がプランに反映される
- `azd-infra-rules` / `azd-hooks` Skill の参照ドキュメントが確認される
- 生成するファイル一覧、各ファイルの役割、ADR との対応関係がプランとしてまとめられる
- プランはセッションフォルダの `plan.md` に保存される (`Ctrl+Y` で閲覧可能)

> [!TIP]
> いきなりコードを生成するのではなく、まずプランを作って進め方を把握する。ADR の内容確認、スコープや実行順序の整理などを行うことで手戻りを減らせる。プランに違和感があれば、この段階で修正できる。

---

### ステップ2: プランの確認と実装

ステップ1でプランが完了すると、Copilot CLI が次のアクションの選択肢を提示する。`Ctrl+Y` でプランを閲覧できる。合意できる内容かを確認する。合意できなければ、Copilot CLIに要望を伝え、プランを仕上げる。

プラン作成後、その後のアクションをCopilot CLIが提案することがある。プランを実行する提案を選択すれば、実装がはじまる。提案がなくとも、「プランを実行して」と指示すればよい。

**期待する結果:**

- プランに従ってコードが生成される
- 推奨構成と ADR の設計判断が Bicep のリソース定義・設定に反映される
- `azd-infra-rules` Skill が適用され、azd 互換の命名・タグ・ネットワーク制約が反映される
- `azure.yaml`（サービス定義・フック設定）と `hooks/` の Python スクリプトも生成される

---

### ステップ3: マルチモデル並列レビュー

> [!NOTE]
> 前のステップでAutopilotモードを選択した場合、ここで通常モードに戻す。`Shift+Tab`でモードは切り替わる。

**Copilot CLIに入力:**

```
/review @infra/ をレビューして。セキュリティ、コスト、運用性の観点で改善点があれば指摘して。Bicep のビルドも実行して。2つのサブエージェント（Opus 4.6 と GPT 5.4）で実行して
```

> [!NOTE]
> このステップでは明示的にレビューを行うが、Copilot CLIは複雑な問題に取り組む際、自律的に別モデルへのレビューを依頼する（rubber-duck エージェント）。2024/4時点ではExperimentalフラグを有効にする必要がある。

レビューで提案された改善点が妥当であれば、受け入れる。

**期待する結果:**

- 2つのサブエージェントが **並列** に起動される（バックグラウンドタスクとして同時実行）
- 異なるモデルで同じコードがレビューされる（観点の違いはモデルや実行ごとに異なる）
- 両方の結果が統合されて報告される
- モデル間で重複する指摘は、信頼度の高い指摘として報告される
- 片方だけが見つけた指摘が、セカンドオピニオンとして報告される

> [!TIP]
> `/model` でモデルを切り替えて何度も実行する必要はない。サブエージェントにモデルを指定して並列実行するため、1回のプロンプトで複数視点のレビューが完了する。

---

## Part 2: Design Snapshot — アーキテクチャ概要の自動生成

### ステップ4: Design Snapshot の生成

**Copilot CLIに入力:**

```
アーキテクチャ概要説明書を作って
```

`architecture-snapshot` カスタムエージェントが自動選択されるはずだが、されない場合は以下で直接選択する。カスタムエージェントとの会話に入るため、作業が終わったら `/agent` で既定に戻すこと。

```
/agent architecture-snapshot
```

**期待する結果:**

- 以下が読み込まれる:
  - `infra/` 配下の Bicep ファイル — 実際の構成
  - `docs/adr/` — 設計判断の根拠
  - `.github/copilot-instructions.md` — プロジェクトの規約
- `docs/architecture-overview.md` としてアーキテクチャ概要説明書が生成される（IaC と ADR から抽出したスナップショット）

> [!TIP]
> アーキテクチャ概要説明書は IaC と ADR からの**スナップショット**であり、手動でメンテナンスしない。IaC や ADR を更新したら、再度実行するだけで最新のアーキテクチャ概要が得られる。

---

## まとめ

- **ADR の設計判断が IaC コードに反映される** — ADR → Bicep への対応を確認
- **マルチモデル並列レビュー** — 1プロンプトで複数モデルの出力を比較
- **azd 構成も ADR から生成** — `azure.yaml` を含む
- **アーキテクチャ概要はスナップショット出力** — IaC + ADR から生成（`architecture-snapshot` エージェント）
