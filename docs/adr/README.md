# Architecture Decision Records (ADR)

このディレクトリには、プロジェクトの設計判断を個別に記録した ADR を格納する。

## ADR とは

ADR は「なぜそうなっているか」を記録するドキュメント。コードは「何を作ったか」を示すが、「何を却下し、なぜその選択をしたか」は示さない。ADR がその隙間を埋める。

## フォーマット

```markdown
# ADR-NNN: タイトル

## Status
Accepted / Superseded by ADR-NNN / Deprecated

## Context
何が問題だったか（2-3行）

## Decision
何を選び、何を却下したか（理由付き）

## Consequences
この判断の結果、何が楽になり、何が制約になるか
```

## 粒度の基準

**「6ヶ月テスト」**: 6ヶ月後に別の人（または自分）が「なぜこうなっている？変えていい？」と聞くか？
- 聞かれそうなら → ADR にする
- 聞かれなさそうなら → 不要（コードや instructions で十分）

**ADR にするもの**: 別の選択をする合理的な理由があった判断
**ADR にしないもの**: 一時的な判断（SKU 選定）、影響の小さい技術的詳細（CIDR レンジ）

## Feature Document からの卒業フロー

```
Feature Document (docs/features/)
  │
  ├── Decisions テーブルの各行を「6ヶ月テスト」で評価
  │
  ├── 通る → docs/adr/NNN-<判断>.md として ADR を作成
  └── 通らない → ADR にしない（コードに反映されていれば十分）
```

## Calibration（いつ Feature Document を作るか）

| シナリオ | Feature Document | 理由 |
|---------|-----------------|------|
| 短い質問、単一ユーティリティ | 不要 | 会話内で完結 |
| 1セッション完結（1時間以下） | 不要（判断があれば ADR のみ） | コンテキスト喪失リスクが低い |
| 複数セッションにまたがる作業 | **作成する** | セッション間の橋渡しが必要 |
| 複数人が関わる作業 | **作成する** | 判断の共有が必要 |
